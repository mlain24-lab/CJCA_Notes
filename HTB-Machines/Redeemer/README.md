# Technical Note: Exploiting Redis Service (Redeemer Machine)

## 1. Overview
The target machine ("Redeemer") was identified with the Redis service running on port 6379/tcp. Redis is an open-source, in-memory data structure store, used as a database, cache, and message broker. It is frequently misconfigured, often lacking authentication, allowing unauthorized access to sensitive data.

## 2. Reconnaissance
Initial port scanning was performed using Nmap to identify open services and service versions.
Command: `nmap -p- -T4 <target_ip>`
Result: Port 6379/tcp (Redis) identified.

## 3. Exploitation Methodology
Two approaches were evaluated to validate the security posture of the Redis instance.

### 3.1 Automated Exploitation (Metasploit)
Used the `redis_extractor` module to confirm database access.
*   **Module:** `auxiliary/gather/redis_extractor`
*   **Procedure:** Set `RHOSTS` to target IP and execute.
*   **Result:** Successfully extracted keys (flag, numb, stor, temp) from database 0.

### 3.2 Manual Interaction (Native Client)
To ensure reliability and minimize the "noise" generated by automated frameworks, the native Redis Command Line Interface was used.
*   **Connection:** `redis-cli -h <target_ip>`
*   **Verification:** Used the `INFO` command to retrieve server statistics and versioning.
*   **Data Extraction:**
    1.  `SELECT 0` (Switching to the primary database).
    2.  `KEYS *` (Listing all available keys).
    3.  `GET <key_name>` (Retrieving the specific content/flag).

## 4. Conclusion
The manual method using `redis-cli` is preferred in professional assessments to reduce the signature footprint of the engagement and provide deeper insight into the protocol's command set. The vulnerability stemmed from an unauthenticated Redis configuration.
